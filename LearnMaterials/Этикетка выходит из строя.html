Этикетка выходит из строя

Уильям Кеннеди

21 ноября 2013 г.

Вы когда-нибудь оказывались в такой ситуации? У вас есть оператор case внутри цикла for, и вы хотите выйти из операторов case и for за один вызов?

var err error
timeout: = time.After (30 * time.Second)

sigChan: = make (chan os.Signal, 1)
signal.Notify (sigChan, os.Interrupt)

complete: = make (chan error)
go launchProcessor (complete )

для {
    выбора {
    случай <-sigChan:
       atomic.StoreInt32 (& shutdownFlag, 1)
       продолжить

    дело <-timeout:
        os.Exit (1)

    случай заблуждается = <-полное:
        перерыв
    }

    // разорвать цикл
    перерыв
}

возвращение ERR

Здесь у меня есть бесконечное ожидание цикла for на трех каналах с использованием оператора select.

Первый случай - это прослушивание события прерывания операционной системы. Если операционная система запрашивает завершение работы программы, в этом случае устанавливается переменная уровня пакета и продолжается возврат в цикл.

Второй случай - это ожидание события тайм-аута. Если программа работает в течение 30 секунд, срабатывает событие тайм-аута, и программа немедленно завершается.

Третий случай - это ожидание полного события. Если Goroutine, запущенный до входа в цикл, завершит свою работу, он уведомит код на этом канале. В этом случае нам нужно выйти как из цикла case, так и из цикла for.

К счастью, нет никакой логики для обработки вне оператора select, поэтому второй оператор break работает. Если бы были другие случаи, которые вырвались из оператора select и не требовали завершения цикла, у меня были бы проблемы. Код потребует больше логики и флагов, чтобы определить, когда выйти из цикла, а когда продолжить итерацию.

У Go есть ответ на этот разделитель кодирования. Вы можете определить метку и перейти к ней.

var err error
timeout: = time.After (30 * time.Second)

sigChan: = make (chan os.Signal, 1)
signal.Notify (sigChan, os.Interrupt)

complete: = make (chan error)
go launchProcessor (complete) )

Цикл:
    for {
        select {
        case <-sigChan:
           atomic.StoreInt32 (& shutdownFlag, 1)
           continue

        case <-timeout:
            os.Exit (1)

        case err = <-complete:
            break Loop
        }
    }

return err

Я немного изменил код, объявив метку Loop чуть выше оператора for. Затем в последнем случае оператору break предоставляется имя этой метки. Этот единственный вызов break переместит выполнение программы за пределы цикла for к следующей строке кода. В этом случае следующая строка кода - это вызов для возврата ошибки.

Вы также можете использовать метку с оператором continue. Это глупый пример, но он показывает вам механизм:

    guestList: = [] string {"bill", "jill", "joan"}
    прибыл: = [] string {"sally", "jill", "joan"}

Контрольный список:
    для _, guest: = диапазон guestList {
        для _, person: = range прибыл {
            fmt.Printf ("Гость [% s] Человек [% s] \ n", гость, человек)

            if person == guest {
                fmt.Printf ("Let% s In \ n", person)
                продолжить CheckList
            }
        }
    }

Вот результат:

Гость [счет] Лицо [sally]
Гость [счет] Лицо [jill]
Гость [счет] Лицо [joan]
Гость [jill] Человек [sally]
Гость [jill] Person [jill]
Let jill In
Гость [joan] Person [sally ]
Гость [joan] Человек [jill]
Гость [joan] Человек [joan]
Впустите Joan

В этом примере есть два цикла for, один вложенный в другой. Из вложенного цикла for оператор continue использует метку для возврата к внешнему циклу for. Из выходных данных вы можете видеть, что
внешний цикл for начинает свою следующую итерацию. После завершения внешнего цикла for выполнение программы продолжается.

Если вы думаете, что это просто причудливый оператор goto, на самом деле это не так. Указанная метка должна включать в себя тот же оператор for, switch или select. Как вы видели, continue все равно начнет следующую итерацию цикла for.

Использование разрывов меток и продолжения в этом сценарии сохраняет код чистым и точным.
